Observability パイプライン構築ガイド

OpenTelemetry Collector を用いたオブザーバビリティ・パイプラインの設計と運用ポイント

2026年2月27日金曜日

なぜパイプラインが必要なのか

クラウド化とマイクロサービス化に伴い、各サービスはメトリクス・ログ・トレースを膨大に生成します。ツールごとにエージェントを配置していると、

  • 収集経路が乱立し障害点が特定しづらい
  • ライセンス課金の上昇（特にログ）
  • 収集フォーマットが統一されずデータの再利用が難しい

といった問題が顕在化します。OpenTelemetry Collector を中心にした observability pipeline を構築すると、収集〜整形〜配信の制御を1箇所に集約できます。

最小構成のアーキテクチャ 

[App] --(OTLP/HTTP)--> [Agent Collector] --(OTLP/gRPC)--> [Gateway Collector] -- fan-out --> [Datadog][Grafana LGTM][S3]
  • Agent Collector: 各ノードにデプロイし、アプリ側の exporter と密結合を避ける。
  • Gateway Collector: VPC 内の1〜数台で受け口を集約し、キューやバッファを持たせる。
  • Backend: SaaS/APM + OSS (Loki, Tempo など) を用途別に使い分ける。

OpenTelemetry Collector の設定例 

receivers:
  otlp:
    protocols:
      http:
      grpc:
processors:
  batch:
  memory_limiter:
    check_interval: 5s
    limit_mib: 512
  attributes:
    actions:
      - key: environment
        value: prod
        action: upsert
exporters:
  loki:
    endpoint: https://logs.example.com/loki/api/v1/push
  otlp/datadog:
    endpoint: https://api.datadoghq.com
    headers:
      DD-API-KEY: ${DATADOG_API_KEY}
service:
  pipelines:
    logs:
      receivers: [otlp]
      processors: [memory_limiter, batch, attributes]
      exporters: [loki, otlp/datadog]

上記の attributes プロセッサで環境タグを付与しておくと、ダッシュボードのフィルタリングが容易になります。memory_limiter は Collector 自体の OOM を防ぐので、本番では必須です。

運用時の観点

  1. バックプレッシャー制御: エクスポーターごとにリトライやキュー長の上限を設定し、停止時のロストを最小化します。
  2. スキーマ互換性: OpenTelemetry Schema 1.0 以降でフィールド名が変わるケースがあるので、otelcol --version と schema URL を定期的に確認します。
  3. コストガードレール: ログの sampling を Collector 側で実装し、一定以上のトラフィックが来た場合は重要タグを含むものだけを通すなど安全装置を持たせます。
  4. 構成のテスト自動化: Collector の設定は単なる YAML なので、otelcol --config config.yaml --dry-run を CI で回すと人為的な typo を早期に検知できます。

データ品質向上の Tips

  • Semantic Conventions の準拠: HTTP サービスなら http.method, http.route など必須属性を埋めることで、可視化ツール側の自動集計が機能します。
  • メタデータの enrich: デプロイ情報（Git SHA、k8s namespace）を Collector 側で付与すると、アプリ改修との相関が取りやすい。
  • PII 対策: filter プロセッサで個人情報をマスキングし、誤送信を防ぐ。

まとめ

Observability pipeline は単にテレメトリデータを転送する装置ではなく、SLO を守るための制御点です。Collector を中心に据えた構成であれば、

  • データの正規化
  • コスト最適化
  • 配信先の抽象化

を同時に実現できます。まずはステージングで Collector を 1 本立て、既存の APM からのルーティングを段階的に切り替えるところから始めてみてください。
